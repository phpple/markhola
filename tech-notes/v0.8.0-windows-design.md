# v0.8.0 Windows 11 技术设计文档

## 一、范围定义

`v0.8.0` 对 Windows 11 的支持，**不能以删除历史已发布能力为代价**。  
本版本的目标范围是：在新增 Windows 11 平台支持的同时，覆盖 `<= v0.7.3` 的全部累计用户能力集合。

这意味着 Windows 11 版本不能是“缩水移植版”。凡是在 `v0.7.3` 及之前已经属于正式能力的内容，`v0.8.0` 都必须在 Windows 上提供等价使用路径。

## 二、本版本的产品规则

1. 保持现有 macOS 行为继续可用。
2. Windows 11 必须补齐 `v0.7.3` 及以前已经发布过的完整用户可见能力。
3. 平台差异只能体现在实现方式上，不能体现为范围缩减。

## 三、Windows 11 必须覆盖的累计能力集合

### 1. 应用启动与壳层能力

- 在 Windows 11 上稳定启动
- 能从命令行打开 Markdown 文件
- 能从资源管理器打开 Markdown 文件
- 支持拖拽 Markdown 文件到窗口中打开
- 保持现有空状态页与状态提示行为

### 2. 文档工作流

- 支持打开 `.md` 与 `.markdown`
- 支持单窗口多文档
- 支持标签页激活、切换、关闭
- 保持每个标签页独立的 dirty 状态与模式状态
- 支持 Save 与 Save As
- 与现有产品一致的从磁盘重新加载行为

### 3. 只读渲染能力

- 标题、链接、图片、表格、列表、引用、代码块
- 语法高亮
- 主流语言高亮增强
- Mermaid 渲染
- 数学公式渲染
- 代码行号与语言徽标

### 4. 可写编辑能力

- 只读 / 可写模式切换
- 编辑器行号
- 全选、复制、粘贴、剪切、撤销、重做
- `Ctrl + A / E` 行首行尾移动
- `Tab / Shift + Tab` 缩进与反缩进

### 5. 查找工作流

- 只读模式页内查找
- 可写模式查找与替换
- 共享查找面板及其快捷键行为

### 6. 输出能力

- 导出 HTML
- 导出 PDF
- 打印当前文档
- 导出 / 打印时必须针对当前激活标签页
- 可写模式下未保存修改也必须进入导出 / 打印结果

### 7. 原生应用外壳能力

- Windows 原生菜单：`File / Edit / Tab / Help`
- 菜单命令与已发布 macOS 应用能力等价
- 内置 Documentation 入口
- About 入口

## 四、当前 Windows 差距

截至当前版本，Windows 相对累计范围仍存在以下明显缺口：

1. 还没有 Windows 原生菜单栏
2. PDF 导出仍然是 macOS-only
3. 打印仍然是 macOS-only
4. 部分文档仍把 Windows 描述为受限路径

以上缺口在当前范围约束下都属于 `v0.8.0` 的 release blocker。

## 五、设计策略

### 1. 保持共享应用核心不分叉

以下逻辑继续保持共享：

- workspace 状态
- 渲染状态
- 编辑器状态
- 文档生命周期
- 查找 / 替换行为
- 导出流程编排入口

平台相关能力应包裹共享核心实现，而不是复制一套高层流程。

### 2. 新增 Windows 原生菜单层

在现有 macOS 菜单实现旁边新增 Windows 菜单模块。

其职责应包括：

1. 构建 Windows 下的 `File / Edit / Tab / Help` 菜单树
2. 尽量复用现有 `UserEvent` 事件映射
3. 仅在现有事件不足时新增最小必要的 `UserEvent`
4. 像 macOS 一样，根据 workspace 状态同步菜单可用性

建议结构：

- 现有 macOS 菜单文件尽量不动
- 在 `src\implementation\app\` 下增加 Windows 专用菜单模块
- 若两端共用标签或派发辅助逻辑，再提取共享层

### 3. 恢复 Windows 输出能力对齐

既然 PDF 导出与打印在 `v0.7.3` 及以前已经属于正式能力，就不能在 Windows 中继续保留 stub 行为。

实现方向：

1. 用真实 Windows 可执行路径替换 PDF stub
2. 用真实 Windows 可执行路径替换打印 stub
3. 尽量复用现有“文档 -> 渲染 HTML -> 输出”的共享准备流程
4. 将平台原生 WebView / 打印 / PDF API 收敛到窄接口中，避免高层逻辑分叉

关键约束：

- Windows 输出必须针对当前文档内容，而不是应用壳层
- 可写模式下未保存修改必须被包含

### 4. 快捷键保持能力对齐、修饰键平台化

命令集合保持一致，修饰键按平台适配：

- macOS 使用 `Command`
- Windows 使用 `Ctrl`

菜单、键盘事件、用户文档必须描述同一组命令，只是修饰键按平台替换。

### 5. 保持 Windows 启动链路稳定

Windows 端当前已经修复了大体积内联 shell 带来的启动问题。后续新增菜单或输出功能时，不能回退或破坏现有启动方案。

### 6. 文档重新与发布范围对齐

当实现补齐后，发布文档不应再描述：

- Windows 无菜单
- Windows 无 PDF 导出
- Windows 无打印

如果能力已经补齐，文档必须同步恢复为等价描述。

## 六、建议的实现切片

### 切片 1：Windows 原生菜单基础设施

- 菜单创建
- 菜单动作派发
- 菜单状态同步

### 切片 2：Windows 菜单命令补齐

- File 命令
- Edit 命令
- Tab 命令
- Help / About 命令

### 切片 3：Windows PDF 导出能力补齐

- 真实导出路径
- 当前标签页校验
- 未保存修改校验

### 切片 4：Windows 打印能力补齐

- 真实打印路径
- 当前标签页校验
- 未保存修改校验

### 切片 5：发布文档回归一致

- README
- 内置帮助文档
- release checklist

## 七、非目标

以下内容当前仍不属于本切片，除非后续单独确认：

- Windows 安装器打包
- 与能力对齐无关的 Windows 视觉重设计
- 超出 `<= v0.7.3` 累计集合的新产品能力
- 未明确确认的 `v0.7.4` 或 `v1.0.0` 范围扩展
