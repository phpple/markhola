# v0.7.6 Technical Design

## Scope

`v0.7.6` focuses on the app shell viewing experience:

1. add multiple shell themes
2. add a new `View` menu before `Help`
3. add menu-driven fullscreen document viewing

## User-visible behavior

### Theme selection

The app will support four shell themes:

- `default`
- `github`
- `dark`
- `light`

The selected theme is changed from:

- `View > Theme > Default`
- `View > Theme > GitHub`
- `View > Theme > Dark`
- `View > Theme > Light`

Theme changes should apply immediately in the current window without reopening the app or the document.

The selected menu item should remain visually selected in the native macOS menu.

### Fullscreen entry

The app will expose a `View > Toggle Full Screen` action.

Expected behavior:

- works with an empty window
- works while previewing a document
- works while editing a document
- does not close the current document
- does not reset the selected theme

## Implementation plan

### Theme assets

Expand `themes/` from a single `default/layout.css` to multiple theme directories:

- `themes/default/layout.css`
- `themes/github/layout.css`
- `themes/dark/layout.css`
- `themes/light/layout.css`

Each theme remains editable as plain CSS and is packaged into the macOS app bundle.

### Runtime theme model

Introduce an app-level theme enum for the shell:

- stable theme id used by menu actions
- human-readable label used by status messages and menus
- directory name used to load the CSS resource

The runtime will keep the currently selected shell theme in memory.

### Theme loading

Generalize shell theme loading so the app can resolve CSS for any supported theme from:

1. repository working directory during development
2. app-adjacent theme directory if present
3. packaged app resources
4. embedded fallback CSS for the default theme

If a non-default theme file is missing, fall back to the default CSS instead of failing the window.

### Live theme application

The shell HTML will expose a dedicated `<style>` node for app theme CSS.

When the selected theme changes:

1. Rust loads the selected CSS
2. Rust sends it to the webview
3. the shell replaces the theme style contents in place
4. the native menu checkmark updates

No document reload is required.

### Native menu structure

Insert a new `View` menu between `Tab` and `Help`.

Menu contents:

- `Theme`
  - `Default`
  - `GitHub`
  - `Dark`
  - `Light`
- separator
- `Toggle Full Screen`

Theme items dispatch dedicated user events instead of overloading the help or file menus.

### Fullscreen handling

Add a dedicated user event for fullscreen toggling.

At runtime:

- if the window is not fullscreen, enter fullscreen
- if the window is already fullscreen, return to windowed mode

This uses the existing app window and should not affect workspace state.

## Validation strategy

### Automated

- unit tests for theme resource loading
- shell tests for theme HTML injection
- app tests for supported theme enumeration and shell output

### Manual

- open the app and switch among all four themes
- verify the current theme stays selected in the menu
- verify theme switching works in readonly mode
- verify theme switching works in writable mode
- verify `View > Toggle Full Screen` enters and exits fullscreen
- verify fullscreen works with a loaded Markdown document
- verify fullscreen works after changing themes
