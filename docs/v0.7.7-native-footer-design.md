# v0.7.7 Technical Design

## Scope

Move the footer status bar out of the WebView and make it a native macOS window component.

## Why

The current footer is rendered inside the HTML shell, so its position and sizing are tightly coupled to the WebView layout. This makes the footer harder to keep visually stable across empty state, readonly mode, writable mode, and future app-shell changes.

## Expected Behavior

1. The footer status bar is no longer rendered by the WebView HTML shell.
2. The footer is attached to the native macOS window layout.
3. The footer stays pinned to the bottom of the window during:
   - empty workspace
   - readonly preview
   - writable editor
   - theme switching
   - tab switching
4. The WebView content area fills the space above the native footer.

## Implementation Notes

1. Create a native container view for the window content on macOS.
2. Place the existing WebView inside that container as the top/fill region.
3. Create a native footer view below it.
4. Render the footer fields natively:
   - path
   - words
   - lines
   - mode
   - status
5. Keep the current workspace state as the single source of truth.
6. Extend the Rust-side presentation/update path so footer values are updated directly in native UI, instead of relying on DOM updates.
7. Remove the HTML footer from `shell_markup.html` after the native footer is in place.
8. Keep the top in-document status text (`#status`) only if it is still useful for transient messages inside the preview header; otherwise remove or simplify it in a follow-up cleanup.

## Non-Goals

- Rebuilding the whole app shell as native UI
- Changing document rendering from WebView to native text/HTML views
- Adding new footer fields in the same change

## Risks

1. The current window bootstrapping is simple (`Window + WebView`), so this change introduces native view composition work.
2. Theme colors currently come from WebView CSS; the native footer will need a matching color strategy.
3. Footer updates currently piggyback on WebView workspace rendering, so update flow must be split carefully.
